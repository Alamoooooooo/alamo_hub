# TODO: Codex Plugin / Marketplace Packaging

This note tracks the work needed to turn `alamo_skillhub` or `memory_hotspot_repair_kernel` into a Codex plugin / marketplace-friendly distribution artifact later.

Status right now:

- not started by choice
- current repo structure is still optimized for package development, not plugin distribution
- this TODO is meant to preserve the next-step plan without changing the product yet

## Goal

Package the memory skill suite so it is easier to install, present, and potentially distribute through Codex plugin and marketplace flows.

The main target is:

- keep `packages/memory_hotspot_repair_kernel/` as the product source
- add a separate plugin distribution layer later
- avoid forcing the whole repo to become a plugin root immediately

## Recommended Direction

Prefer this split:

```text
alamo_skillhub/
├── packages/
│   └── memory_hotspot_repair_kernel/
└── plugins/
    └── memory-hotspot-repair/
        ├── .codex-plugin/
        │   └── plugin.json
        ├── skills/
        ├── assets/
        └── README.md
```

Reason:

- `packages/` stays as the development and product structure
- `plugins/` becomes the distribution structure
- future packages can choose independently whether they become installable plugins

## TODO List

### 1. Decide scope

- Decide whether the first plugin is:
  - only `memory_hotspot_repair_kernel`
  - or a broader `alamo_skillhub` catalog
- Decide whether the first release is:
  - `skills-only`
  - or later `app-plus-skills`

Current recommendation:

- start with one `skills-only` plugin for `memory_hotspot_repair_kernel`

### 2. Create plugin distribution folder

- Add `plugins/memory-hotspot-repair/`
- Add `.codex-plugin/plugin.json`
- Add `README.md`
- Add `assets/` with icon and screenshot placeholders

### 3. Define plugin manifest

Prepare `plugin.json` with at least:

- stable plugin name
- version
- concise description
- skills directory path

Draft positioning:

- plugin name: `memory-hotspot-repair`
- short description: portable AI coding-agent skills for memory hotspot detection and bounded optimization loops

### 4. Map package skills into plugin skills

- Decide whether plugin `skills/` should:
  - copy files from `packages/memory_hotspot_repair_kernel/`
  - symlink them
  - or be generated by a small packaging script

Current recommendation:

- use a packaging script or generated copy, not manual duplication

### 5. Add plugin-facing docs

Write a plugin README that directly answers:

- what problem this plugin solves
- when to use it
- what the main entry point is
- how Codex users install it
- what outputs the workflow produces

Keep this shorter and more install-oriented than the repo root README.

### 6. Prepare marketplace assets

- plugin icon
- optional screenshots
- short one-line summary
- starter prompts
- minimal test cases or demo scenarios

Starter prompt candidates:

- `Use $memory-optimizer-agent to run one bounded memory optimization pass on this codebase.`
- `Use $memory-check to produce a structured memory hotspot report for this repository.`
- `Use $memory-fix to remediate the highest-priority memory bottleneck while preserving behavior.`

### 7. Clarify packaging workflow

- Decide how plugin artifacts are built from `packages/`
- Add a small script if needed
- Document how to refresh plugin `skills/` after package changes

Useful outcome later:

- one command to rebuild the plugin distribution folder from the package source

### 8. Validate local install flow

Later, verify:

- local plugin install works
- installed skills appear correctly
- entry-point prompts are readable
- package metadata and plugin metadata do not drift

### 9. Optional future expansion

If the first plugin works well, consider:

- multiple plugins under `plugins/`
- one curated marketplace repo
- host-specific variants if another platform requires a different wrapper
